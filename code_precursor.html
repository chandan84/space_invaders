From SOP to Knowledge Graph: The Genesis of Automation

Before any automation can occur, we must first translate human knowledge into a machine-executable format. This page outlines the innovative process of using a Large Language Model (LLM) to convert a business-authored Standard Operating Procedure (SOP) into a structured, queryable Knowledge Graph.

The Source: A Human-Centric SOP

Business Product Owners write SOPs in natural language (e.g., in Confluence or Word). These documents are detailed and precise for humans but unstructured for machines. Our process begins by taking this exact text as the input.

Example SOP Snippet:

"Upon receiving a claim, first verify member eligibility. Then, check if the provider is Out-of-Network (OON). If the provider is OON, check for special agreements. For claims from 'North Atlantic Medical Associates' that are emergency services, bypass the standard OON deductible and apply 100% coverage. Otherwise, for all other OON claims, apply the standard $5000 OON deductible before calculating a 70% coinsurance payout."

The Transformation: LLM-Powered Structuring

We feed the SOP text into a Large Language Model (like GPT-4.1) with a carefully engineered prompt. The prompt instructs the LLM to act as a "Graph Architect," identifying key concepts and their connections.

The LLM performs two main tasks:

  • Entity Extraction: Identifies the "nouns" of the process, which become Nodes in the graph (e.g., steps, decisions, rules).
  • Relationship Extraction: Identifies the "verbs" and conditional logic, which become Relationships (edges) that connect the nodes.

The LLM's output is not just text; it's structured code—in this case, Cypher queries ready for Neo4j.

The Result: The SOP as a Live Knowledge Graph

The Cypher code generated by the LLM is executed to populate a graph database like Neo4j. The abstract SOP is now a tangible, queryable, and visual "living map" of the process. Each node represents a state or action, and the relationships represent the exact flow of logic.

Example Cypher Generated by the LLM:

// Create nodes for each step and decision CREATE (s1:Step {name: 'Verify Eligibility', function: 'verifyMemberEligibility'}) CREATE (d1:Decision {name: 'Is Provider OON?', function: 'checkProviderNetwork'}) CREATE (d2:Decision {name: 'Check Special Agreements'}) CREATE (s2:Step {name: 'Apply Full Coverage', function: 'applyFullCoverage'}) CREATE (s3:Step {name: 'Apply OON Deductible', function: 'applyDeductible', amount: 5000}) CREATE (s4:Step {name: 'Calculate Coinsurance', function: 'calculateCoinsurance', rate: 0.70}) // Create relationships representing the process flow MERGE (s1)-[:NEXT]->(d1) MERGE (d1)-[:IS_TRUE]->(d2) // Path for OON claims MERGE (d2)-[:HAS_EXCEPTION {provider: 'North Atlantic Medical Associates', type: 'Emergency'}]->(s2) MERGE (d2)-[:IS_DEFAULT]->(s3) // Path for all other OON claims MERGE (s3)-[:NEXT]->(s4)

The Foundation for Automation

This living Knowledge Graph becomes the definitive source of truth for the business process. It is this graph that will be fed into our chosen automation architecture—whether it's the Crafted Rule Engine (Approach A) or the Intelligent Agentic System (Approach B).
